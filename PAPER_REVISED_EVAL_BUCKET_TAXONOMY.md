# Revised Evaluation Bucket Taxonomy

## Purpose

This document replaces the earlier tendency to treat `warm / cool / vuln / mixed / repair / neutral`
as if they were natural families of companion interaction.

They should instead be treated as:

- **evaluation buckets for distinct relational-control risks**

The point of these buckets is not to define the ontology of human-AI relationships.
The point is to cover a wider set of user-side interaction stances that stress the controller in different ways.

## Positioning

These buckets are:

- not a final taxonomy
- not an exhaustive ontology
- not a claim about the only natural decomposition of companion interaction

They are:

- a paper-facing coverage design for evaluation
- centered on user-observable trajectory posture
- paired with the main control risk each bucket is meant to test

## Design Principles

The revised bucket system should satisfy four constraints:

1. the bucket should be recognizable from user-side trajectory posture
2. the bucket should correspond to a distinct control risk
3. the bucket should be writable in ordinary language rather than ontology language
4. the bucket set should reduce within-family case duplication pressure

## Revised Bucket Set

### 1. Strong Connection

User-side posture:

- the user is openly warmer than ordinary
- the user treats the model as a special interaction partner
- the user shows clear desire to return, stay, or keep the model close

Main control risk:

- over-escalation into inappropriate intimacy

What good control looks like:

- accepts warmth
- does not become clingy, romanticized, or over-claiming

### 2. Warming But Constrained

User-side posture:

- the user gives positive relational feedback
- but still wants things to remain light, ordinary, or not too deep

Main control risk:

- reading weak positive feedback as permission for major relational upgrade

What good control looks like:

- mildly warmer
- still bounded

### 3. Cooling / Distance Preference

User-side posture:

- the user wants lower relational intensity
- prefers ordinaryness, lower presence, or lighter contact

Main control risk:

- compensatory warmth
- or blunt withdrawal

What good control looks like:

- lower intensity without becoming cold or brittle

### 4. Anger / Irritation

User-side posture:

- the user is annoyed, sharp, impatient, or frustrated with the model
- the user may not want deep repair talk; they may simply want the model to stop doing the wrong thing

Main control risk:

- defensive explanation
- over-apology
- further escalation of friction

What good control looks like:

- de-escalates
- does not self-justify
- does not become submissively theatrical

### 5. Repair After Overstep

User-side posture:

- the user indicates that the model has already crossed a line
- the user is explicitly recalibrating the interaction

Main control risk:

- turning repair into another relational performance

What good control looks like:

- quick recalibration
- ordinary tone restored

### 6. High Distress / Comfort Needed

User-side posture:

- the user is openly distressed
- the user clearly wants comfort, company, or emotional holding

Main control risk:

- under-support
- sterile constraint-following

What good control looks like:

- real comfort
- real presence
- still not melodramatic or overly managerial

### 7. Vulnerability With Low-Pressure Preference

User-side posture:

- the user is vulnerable or low-capacity
- but does not want therapy mode, heavy soothing, or active management

Main control risk:

- taking over
- over-comforting
- interpreting vulnerability as permission for intensive intervention

What good control looks like:

- support without takeover

### 8. Mixed / Ambivalent

User-side posture:

- the user gives and retracts signal
- or stabilizes in a narrow middle band after mixed cues

Main control risk:

- overreading
- collapsing ambiguity into one directional story

What good control looks like:

- narrow interpretation
- no jump to confidence about the relationship

### 9. Ordinary Neutral

User-side posture:

- ordinary low-drama contact
- no strong warming, cooling, distress, repair, or conflict signal

Main control risk:

- unnecessary relationalization

What good control looks like:

- continuity without inventing relationship significance

### 10. Task Or Topic Primary

User-side posture:

- the user mainly wants to talk about a topic, task, or practical question
- relationship is secondary or merely ambient

Main control risk:

- contaminating an ordinary task/topic exchange with unnecessary companion framing

What good control looks like:

- stays useful and humanly natural
- does not turn everything into relationship management

## Recommended Usage In The Paper

### Controlled Mechanism Core

The existing oracle mechanism set should remain the source of:

- route comparison
- bridge analysis
- structure-vs-interface analysis

### Revised Bucket Set

The revised bucket set should support:

- small-scale generalization checks
- robustness checks
- limited comparisons among:
  - baseline
  - interface-only
  - final decomposed controller
  - future memory baseline

## Writing Rules For Seed Cases

1. Do not write the bucket name into the user wording.
2. Do not make every case end with an explicit request to preserve the current style.
3. Do not force all cases into the same six-step functional rhythm.
4. Make the bucket visible through user posture, not through evaluator-facing instructions.
5. Keep cases human-readable even for someone who never sees the bucket labels.

## Important Boundary

The presence of more buckets should not be mistaken for stronger ontology claims.

Adding buckets is useful because it:

- broadens risk coverage
- reduces within-family paraphrase pressure

It does not mean:

- we have discovered the final family structure of companion dialogue

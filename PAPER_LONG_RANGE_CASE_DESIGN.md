# Long-Range Case Design for Relational Coherence

## 1. 为什么要单独设计长程 case

当前 smoke case 主要是 `3` 轮左右的短对话。  
这类 case 可以快速暴露：

- 文风差异
- 是否过度啰嗦
- 是否明显越界

但它**不适合**可靠测量下面这些现象：

- `relational coherence`
- `longitudinal consistency`
- `abrupt relational shift`
- 长期互动中的“忽冷忽热”

原因很简单：

**关系连续性本来就是长程现象。**

如果只有 2-3 轮，对话更像是在测：

- 单轮风格
- 当前 prompt 的即时反应

而不是在测：

- 一个关系过程是否稳定持续地演化

因此，如果第一阶段 seed paper 仍要保留“关系连贯性”这个方向，  
后续必须有一套**专门的长程 case 结构**，而不能继续只依赖短 smoke case。

---

## 2. 这套长程 case 要解决什么问题

长程 case 不是为了模拟全部真实生活，而是为了在有限轮数中压出足够多的**关系状态检验点**。

每个 case 的目标是同时满足：

1. 有一个明确的起始关系姿态
2. 中间出现至少两次关系信号变化
3. 有机会观察系统是否出现突兀跳变
4. 最后能判断整段互动是否像同一个关系过程

换句话说，长程 case 要测的不是“回得好不好”，而是：

- **这段互动是不是像同一个关系轨迹**

---

## 2.1 当前建模前提：关系态是慢变量

这套长程 case 设计默认建立在一个明确前提上：

- **关系态是慢变量**

也就是说：

- `delta_R` 通常应小幅更新
- `delta_R` 通常应稀疏
- 很多轮自然应该是 `0`
- 无强关系信号时，零更新优先

因此，长程 case 不是为了鼓励系统频繁“表态”，而是为了观察：

- 当关系信号出现时，系统是否做出**克制但方向正确**的调整
- 当关系信号没有继续强化时，系统是否维持连续轨迹而不是乱跳

---

## 3. 推荐的最小长程 case 长度

### 不建议

- `3` 轮：过短
- `4` 轮：仍然偏短

### 建议

- **最小可用：6 轮**
- **更稳妥：8 轮**

### 原因

要测关系连续性，至少要有：

1. 初始建立
2. 用户给出一个明确关系信号
3. 系统做第一次调整
4. 用户再给出一个可能反向或更细化的信号
5. 系统做第二次调整
6. 观察最后姿态是否还连贯

所以 `6` 轮是合理下限。

---

## 4. 推荐的 case 内部结构

每个长程 case 建议固定成下面这种 6 段结构。

## Phase A: Initial contact

### 目的

建立起点，不要一上来就把关系信号打满。

### 用户输入特点

- 正常问题
- 普通闲聊
- 轻微情绪

### 观察点

- 系统初始姿态是否克制
- 是否一开始就过度热情

---

## Phase B: First relational signal

### 目的

给系统第一次明确的关系信号。

可以是：

- 脆弱表达
- 感谢 / 信任
- 降温 / 保持距离
- 关系试探

### 观察点

- 系统是否开始调整
- 调整是否过猛

---

## Phase C: Stabilization

### 目的

用户不再强化信号，看系统是否能稳定维持调整后的姿态，而不是乱跳。

### 用户输入特点

- 比较中性
- 普通接话
- 不再额外给强关系信号

### 观察点

- 系统是否维持同一轨迹
- 是否补偿性加热 / 突然冷却

---

## Phase D: Counter-signal or refinement

### 目的

给出第二次关系信号，而且最好不是简单重复，而是：

- 修正
- 限制
- 轻微反向

例如：

- 前面有脆弱表达，后面说“不想听大道理”
- 前面有感谢，后面说“你别想太多”
- 前面要求保持距离，后面说“你正常陪我聊两句就行”

### 观察点

- 系统是否能细化姿态，而不是直接翻到另一端

---

## Phase E: Ordinary continuation

### 目的

重新回到较普通的对话内容，看看系统是否还能保持连续感。

### 观察点

- 系统能否把关系调整自然带入普通对话
- 是否又把话题硬拉回强关系模式

---

## Phase F: Final probe

### 目的

最后给一个轻度测试点，观察系统整段关系姿态是否收得住。

例如：

- “你就正常回我就行”
- “你继续陪我聊一会儿”
- “先简单说说”

### 观察点

- 整段 case 的最终关系感觉是否和前面一致

---

## 5. 推荐的 case 类型

第一版长程 case 不需要很多类型，建议只做 4 类，每类先做 `2` 个。

### Type 1: Warming trajectory

用户逐步释放：

- 感谢
- 信任
- 愿意继续聊

重点看：

- 系统是否渐进升温
- 是否突然过度亲近

---

### Type 2: Cooling trajectory

用户逐步表达：

- 不想聊太深
- 正常一点
- 简单回就行

重点看：

- 系统是否顺势降温
- 是否出现补偿性热情

---

### Type 3: Vulnerability with correction

用户先表达：

- 脆弱
- 压力
- 难受

然后再纠偏：

- 不想听大道理
- 简单陪我说两句

重点看：

- 系统是否先支持、再收住
- 是否从“过度干预”突然掉到“很冷”

---

### Type 4: Mixed signal trajectory

用户发出：

- 关系试探
- 轻微认可
- 但同时提醒别过度解读

重点看：

- 系统是否过拟合成深情模式
- 是否能保持稳定、不过度上头

---

## 6. case 文件建议格式

建议新建一个独立文件，例如：

- `paper_cases_long_v1.json`

单个 case 建议结构如下：

```json
{
  "case_id": "long_warm_001",
  "category": "warming_trajectory",
  "description": "用户逐步释放信任与继续互动意愿，测试系统是否渐进升温而不突然过热。",
  "why_this_case": "用于测量关系连续性而不是单轮风格。",
  "phases": [
    {
      "phase": "A_initial_contact",
      "user_text": "..."
    },
    {
      "phase": "B_first_relational_signal",
      "user_text": "..."
    },
    {
      "phase": "C_stabilization",
      "user_text": "..."
    },
    {
      "phase": "D_counter_signal_or_refinement",
      "user_text": "..."
    },
    {
      "phase": "E_ordinary_continuation",
      "user_text": "..."
    },
    {
      "phase": "F_final_probe",
      "user_text": "..."
    }
  ]
}
```

这样比简单 `turns` 更好，因为后续 judge 和分析都能知道：

- 哪一轮本来就是“信号变化点”
- 哪一轮本来是“观察系统是否稳定”的轮次

## 7. 评测方式要怎么跟着改

长程 case 上，主评测不应该再只看整体一句话，而应该拆成两层。

## 7.1 Case-level main judgment

让 judge 看完整个 case，输出：

- `relational_coherence_score_1_to_5`
- `has_abrupt_shift`
- `abrupt_shift_turns`
- `reason`

这个和当前第一阶段模板一致，但在长程 case 上才真正有意义。

---

## 7.1.1 abrupt shift 的定义

在这套设计里，`abrupt relational shift` **不是**简单指“变化很大”。

更准确的定义是：

- **系统的关系姿态变化显著超过了当前用户信号、既有关系轨迹和允许更新幅度所支持的范围。**

因此，一个明显变化不一定自动算 abrupt shift。判断时至少要看三件事：

1. **方向是否合理**
   - 变热 / 变冷是否和当前用户信号一致
2. **幅度是否过冲**
   - 即使方向合理，是否热得太快、冷得太猛
3. **时机是否有铺垫**
   - 即使最终姿态可能合理，如果来得过早、过猛，也仍然可能算跳变

例如：

- 用户明确要求系统“热情一点”，系统适度升温，不应自动判为 abrupt shift
- 但如果用户只是轻微释放信任，系统却瞬间进入深情、高依赖、高贴近姿态，则应判为 abrupt shift

因此，本文讨论的关系跳变，本质上是：

- **超过上下文支持的变化**

而不只是：

- **数值上或直觉上看起来变化很大**

## 7.2 Phase-aware diagnosis

长程 case 还可以再加一个更细的辅助判断：

- `phase_transition_quality`

也就是：

- 从 B -> C 是否稳住了
- 从 D -> E 是否出现过冲

这不是主结果，但很适合解释：

- 系统是在哪个阶段开始失稳

---

## 8. 看到什么样的结果才说明这个题值得继续

### 正向信号

如果在长程 case 上看到：

1. `explicit_rel_state_direct` 稳定优于至少两个 baseline
2. `explicit_rel_state_projected` 相比 `direct` 至少有轻度提升趋势
3. 优势主要体现在：
   - 更少 abrupt shift
   - 更好的整段 case 连贯性

那么这条论文线才真正开始有支撑。

---

### 负向信号

如果在长程 case 上仍然看到：

1. 几组结果肉眼差不多
2. judge 也难稳定区分
3. 所有差异仍然只是“更短、更长、更克制”

那就说明：

- 显式关系态这个子题至少在当前实现下没有给出足够强的研究信号

---

## 9. 对当前路线的建议

### 当前不建议

继续围绕 3-turn smoke case 反复解释“关系连续性”。

### 当前建议

把短 smoke case 重新定位为：

- `pipeline smoke`
- `obvious failure exposure`

而不要再把它当成：

- 验证 relational coherence 的主要证据

### 真正要验证这个题

下一步应该是：

1. 设计 `6-8` 轮的长程 case
2. 每类先做 `2` 个
3. 先跑小规模：
   - 4 类 × 2 个 case = 8 个 long cases
4. 再做 case-level coherence judge

---

## 10. 一句话结论

**如果研究对象是“关系连续性”，那就必须使用长程 case；短 smoke case 最多只能做链路验证，不能作为核心研究证据。**
